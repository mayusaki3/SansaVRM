# トレーサビリティ運用方針

## 1. 目的

本ドキュメントは、SansaVRM の仕様・テスト仕様・実装コード間の対応関係を管理するための運用方針を定義する。

本方針は、HLDocS の Traceability 規約に基づき、実装開始前に以下を明確化することを目的とする。

- 仕様とテスト仕様の対応関係
- テスト仕様とコードの対応関係
- `sec_id` の確定順序
- 実装中の暫定トレース方法
- HLDocS 化後の正式トレース方法
- 未参照・孤立参照の検査方針

---

## 2. 基本方針

SansaVRM では、仕様・テスト仕様・コードの対応関係を以下の順序で確定する。

1. テスト仕様で検証単位を定義する
2. テスト仕様側で `sec_id` を確定する
3. 対応する仕様へ `sec_id` を反映する
4. 実装コードへ `@hldocs.ref` を付与する
5. 仕様・テスト仕様・コードの参照整合性を検査する

`sec_id` は、仕様単体では確定しない。

---

## 3. 運用段階

## 3.1 HLDocS 化前

HLDocS 形式への整備が完了するまでは、正式な `doc_id#sec_id` 参照は使用しない。

この段階では、コードコメントに暫定トレースを記載してよい。

例：

```rust
// TODO(trace): CoreAPI仕様 / Model管理API / create_model に対応
```

---

## 3.2 HLDocS 化後

HLDocS 化により、各ドキュメントに以下が整備された後、正式なトレースを付与する。

- LLM-MANAGED ブロック
- `doc_id`
- testspec 起点で確定した `sec_id`
- spec / testspec 間の対応関係

コード側では以下の形式を用いる。

```rust
// @hldocs.ref doc-XXXXXXXXXXXX#sec_xxxxxxxx
```

---

## 4. sec_id 運用

## 4.1 確定主体

`sec_id` は、テスト仕様（testspec）を起点として確定する。

仕様（spec）側で先に `sec_id` を生成してはならない。

---

## 4.2 形式

`sec_id` は以下の形式とする。

```text
sec_<random>
```

以下を含めてはならない。

- 意味語
- 章名
- 番号
- テスト番号
- document_type
- 用途を示す語

---

## 4.3 対応単位

`sec_id` は、以下のような検証単位に対して付与する。

- JSON Schema の構造検証単位
- Validator の検証規則単位
- Core API の操作単位
- 変換処理のマッピング単位
- ラウンドトリップの検証単位

---

## 5. コード側トレース

## 5.1 暫定コメント

HLDocS 化前は、正式IDを使わず、仕様名・章名・処理名で暫定的に対応関係を示す。

例：

```rust
// TODO(trace): Validator実装仕様 / ID一意性検証 に対応
```

---

## 5.2 正式コメント

HLDocS 化後は、`doc_id#sec_id` を使用する。

例：

```rust
// @hldocs.ref doc-20260424-000001Z-abcd#sec_x7k2m9p1
```

---

## 5.3 コメント配置

`@hldocs.ref` は、対応する処理単位の直前に記載する。

例：

```rust
// @hldocs.ref doc-20260424-000001Z-abcd#sec_x7k2m9p1
pub fn create_model(input: CreateModelInput) -> Result<Model> {
    ...
}
```

---

## 6. トレース対象

以下をトレース対象とする。

- Core data model
- Validator
- Core API
- Importer / Exporter
- JSON Schema
- テストコード

以下は原則としてトレース対象外とする。

- README
- サンプルデータ
- CI設定
- 開発補助スクリプト
- 一時的な実験コード

---

## 7. 検査方針

実装完了前に、以下を検査する。

- testspec に存在する `sec_id` が spec に反映されていること
- spec に存在する `sec_id` が testspec 由来であること
- コード上の `@hldocs.ref` が実在する `doc_id#sec_id` を参照していること
- 未参照の testspec が存在しないこと
- 孤立したコード参照が存在しないこと
- 参照先不明の `@hldocs.ref` が存在しないこと

---

## 8. 実装開始時の暫定ルール

HLDocS 化が完了するまでは、以下の暫定ルールで実装を進める。

- 正式な `@hldocs.ref` は付与しない
- `TODO(trace):` による暫定対応を許可する
- 対応仕様名・章名・処理名を記載する
- 後工程で一括して `@hldocs.ref` に置換する

---

## 9. HLDocS 化後の移行手順

1. testspec に `sec_id` を付与する
2. spec に対応する `sec_id` を反映する
3. 暫定コメントを検索する
4. `TODO(trace):` を `@hldocs.ref` に置換する
5. トレース検査を実行する
6. 未解決参照を修正する

---

## 10. 非スコープ

本ドキュメントでは以下を対象外とする。

- `sec_id` の自動生成アルゴリズム
- トレース検査ツールの実装
- HLDocS 形式への全文変換手順
- CI における自動トレース検査の詳細

---

## 11. 結論

SansaVRM では、実装開始時点では暫定トレースを使用し、HLDocS 化後に正式な `doc_id#sec_id` 参照へ移行する。

これにより、実装を止めずに進めつつ、後工程で仕様・テスト仕様・コードの恒久的な紐づけを確立する。
