# テスト雛形作成

## 1. 目的

[importテスト設計](./01_importテスト設計.md) の T-F5-001 ～ T-F5-030 に対応するテスト雛形を作成し、VRM import 実装の進捗をテスト単位で管理可能にする。

---

## 2. 対象ブランチ

- develop

---

---

---

## 5. テスト作成ルール

### 5.1 テスト番号保持

各テストには Step F-5 のテスト番号をコメントとして付与する。

例：

```rust
// T-F5-001: vrm0_import_minimal
#[test]
fn vrm0_import_minimal() {
    // TODO: implement
}
```

---

### 5.2 初期状態

未実装機能に対応するテストは `#[ignore]` ではなく、失敗する形で作成する。

理由：

- 未実装箇所をCI上で可視化するため
- 実装完了時にPASS化を確認するため

---

### 5.3 fixture方針

fixture は最初はコード生成方式とする。

理由：

- binary fixture を増やさない
- 差分レビューを容易にする
- 最小構造をテストごとに明示できる

## 7. 初期テスト実装方針

### 7.1 既存実装でPASS可能な候補

以下は既存実装でPASS可能性がある。

- T-F5-001：vrm0_import_minimal
- T-F5-002：vrm0_import_humanoid
- T-F5-007：vrm0_import_meta
- T-F5-008：vrm1_import_minimal
- T-F5-009：vrm1_import_humanoid
- T-F5-014：vrm1_import_meta

ただし、validator連携、必須bone条件、エラー化が未整備のため、実行結果で確認する。

---

### 7.2 未実装で失敗想定のテスト

以下は現状では失敗想定とする。

- Material / Texture
- Expression / BlendShape
- passthrough
- springBone
- node constraint
- roundtrip unknown fields
- validator連携明示実行
- 異常系エラー化

---

## 8. 実装後確認コマンド

```bash
cargo test -p sansavrm-vrm
```

---

## 9. 完了条件

以下を満たした場合、Step F-8 完了とする。

1. T-F5-001 ～ T-F5-030 のテスト関数が存在する
2. 各テストにテスト番号が付与されている
3. fixture生成ヘルパーが存在する
4. 既存実装でPASSするテストと未実装でFAILするテストが区別できる
5. `cargo test -p sansavrm-vrm` で未実装箇所が確認できる
