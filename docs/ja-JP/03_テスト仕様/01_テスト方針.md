# テスト方針

## 1. 目的

本仕様は、SansaVRM におけるテストの方針および全体戦略を定義する。

本仕様の目的は以下とする。

- 仕様の正当性を検証する
- 実装の品質および安定性を担保する
- 非可逆変換や例外ケースの挙動を明確化する
- 実装と仕様の乖離を早期に検出する
- 自動テスト可能な構造を定義する

---

## 2. テスト対象

本テスト仕様は以下を対象とする。

- JSON Schema 検証
- Validator
- Core API
- 変換処理
- ラウンドトリップ整合性

以下は対象外とする。

- Runtime（レンダリング・物理）
- UI
- ネットワーク
- 外部サービス連携

---

## 3. テストレベル

テストは以下のレベルで実施する。

### 3.1 単体テスト（Unit Test）

対象：

- JSON Schema
- Validator
- Core API 各関数

目的：

- 個別機能の正当性確認

---

### 3.2 結合テスト（Integration Test）

対象：

- Core API + Validator
- Importer / Exporter

目的：

- モジュール間の整合性確認

---

### 3.3 システムテスト（System Test）

対象：

- 変換処理全体
- モデル生成〜出力までの一連処理

目的：

- 実運用に近いシナリオの確認

---

### 3.4 ラウンドトリップテスト（Round-trip Test）

対象：

- 外部フォーマット ⇄ SansaVRM

目的：

- 情報保持率の確認
- 非可逆部分の検証

---

## 4. テスト方針

### 4.1 仕様準拠

すべてのテストは仕様に準拠して作成する。

---

### 4.2 網羅性

以下を必ずカバーする。

- 正常系
- 異常系
- 境界値
- 非可逆変換
- 部分モデル（allow_partial）  
  allow_partial の場合でも参照整合性は維持されること

---

### 4.3 自動化

- すべてのテストは自動実行可能とする
- CI に組み込むことを前提とする

---

### 4.4 再現性

- 同一入力は同一結果を返すこと
- ID生成の安定性を前提とする

---

### 4.5 独立性

- テストは相互依存しない
- 単独実行可能とする

---

## 5. テストデータ

### 5.1 基本データ

- 最小構成モデル
- 標準構成モデル
- 複雑構成モデル

---

### 5.2 異常データ

- 不正ID参照
- 欠損データ
- 型不一致
- 循環接続
- 無効なStateAction

---

### 5.3 外部フォーマット

- glTF サンプル
- VRM サンプル
- URDF サンプル

---

## 6. 成功条件

テストは以下を満たした場合に成功とする。

- JSON Schema 検証を通過する
- Validator が期待通りの結果を返す
- Core API が仕様通り動作する
- 変換結果が期待と一致する
- 非可逆変換が diagnostics に記録される
- Validator のエラーコードおよび severity が仕様通りであること
- diagnostics の内容が仕様 [変換テスト仕様](./05_変換テスト仕様.md) と一致すること

---

## 7. 失敗条件

以下の場合、テストは失敗とする。

- 検証エラーの未検出
- 誤検出（false positive）
- API の不正動作
- データ破損
- 変換結果の不整合

---

## 8. テストケース設計

各テストケースは以下の要素を持つ。

- テストID
- テスト目的
- 入力データ
- 前提条件
- 実行手順
- 期待結果

---

## 9. テスト番号規則

テストIDは以下の形式とする。

```text
[カテゴリ]-[連番]
```

例：

- SCHEMA-STRUCT-001
- SCHEMA-TYPE-002

- VALIDATOR-REF-001
- VALIDATOR-CONN-002
- VALIDATOR-STATE-003

- CORE-MODEL-001
- CORE-MODULE-002
- CORE-SLOT-003
- CORE-CONN-004
- CORE-STATE-005

- CONVERT-GLTF-001
- CONVERT-VRM-002
- CONVERT-URDF-003

- ROUNDTRIP-GLTF-001
- ROUNDTRIP-VRM-002
- ROUNDTRIP-URDF-003

---

## 10. テスト環境

- JSON Schema Validator
- Core API 実装（Rust）
- テストランナー（例：pytest / cargo test）
- CI 環境（GitHub Actions 等）

---

## 11. 非可逆変換の扱い

- 情報が失われる場合は error ではなく warning とする
- `diagnostics` に必ず記録されること
- ラウンドトリップで差分が検出されること

---

## 12. 将来拡張

- パフォーマンステスト
- ストレステスト
- 大規模モデル検証
- 分散環境テスト

---

## 13. 結論

本テスト方針により、以下を保証する。

- 仕様準拠
- 実装品質
- データ整合性
- 変換の信頼性

これにより、SansaVRM は安定した実装基盤として成立する。
