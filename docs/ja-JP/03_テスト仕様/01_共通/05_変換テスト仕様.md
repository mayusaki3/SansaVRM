# 変換テスト仕様

## 1. 目的

本仕様は、[変換仕様](../../02_仕様/01_共通/07_変換仕様.md) に基づく変換処理（Importer / Exporter）の検証テスト仕様を定義する。

本仕様の目的は以下とする。

- 外部フォーマットと SansaVRM Core 間の変換が正しく行われることを確認する
- 構造・接続・状態・属性が正しくマッピングされることを検証する
- 非可逆変換が適切に扱われることを保証する
- `source_raw` および `diagnostics` の動作を検証する
- MuJoCo が標準ファイル変換対象ではなく標準連携対象として扱われることを検証する
- SansaVRM-MuJoCo-Adapter が SansaVRM 本体の公開 API を経由して情報取得することを検証する
- `io_scope`、`mjcf_mapping`、`adapter_artifact` に基づき MJCF 出力対象と Adapter 側補助成果物対象が判定されることを検証する

---

## 2. テスト対象

以下の変換処理を対象とする。

- glTF → SansaVRM（Importer）
- VRM → SansaVRM（Importer）
- URDF → SansaVRM（Importer）
- SansaVRM → glTF（Exporter）
- SansaVRM → VRM（Exporter）
- SansaVRM → URDF（Exporter）

MuJoCo については標準ファイル変換対象ではなく、標準連携対象として以下を対象とする。

- SansaVRM → SansaVRM-MuJoCo-Adapter 連携
- SansaVRM API 経由での Module / Connection / Slot / Property / custom parameter 取得
- 登録スキーマに基づく MJCF 出力対象判定
- 登録スキーマに基づく Adapter 側補助成果物対象判定
- conversion report / diagnostics への記録

---

## 3. テスト分類

---

### 3.1 Import（外部 → SansaVRM）

対象：

- glTF / VRM / URDF

検証内容：

- Model 構造が正しく生成されること
- Module / Slot / State が正しくマッピングされること
- ID が一意かつ安定して生成されること
- 接続（connections）が正しく構築されること
- property_type が正しく設定されること

---

### 3.2 Export（SansaVRM → 外部）

対象：

- glTF / VRM / URDF

検証内容：

- 外部フォーマットとして正しい構造が生成されること
- スキーマ（glTF / VRM / URDF）に準拠していること
- 必須要素が欠落していないこと
- extension が正しく出力されること
- property_type に基づき geom / actuator / sensor が生成されること
- property_type 未設定時のフォールバックテスト
- VRM 0.x / 1.0 の差異が正しく反映されること

---

### 3.3 マッピング検証

対象：

- Module ↔ node / link
- Slot ↔ joint / node関係
- State ↔ animation / expression
- Property分類（property_type → geom / actuator / sensor）
- joint Connection のみが MuJoCo に変換されること

検証内容：

- 双方向で整合すること
- 不整合が発生しないこと

---

### 3.4 ID安定性検証

検証内容：

- 同一入力から同一IDが生成されること
- 複数回実行しても変化しないこと

---

### 3.5 非可逆変換

検証内容：

- 失われる情報が `source_raw` に保存されること
- `diagnostics` に warning が記録されること
- error にならないこと（仕様に従う）
- MJCF へ直接出力できない情報が `adapter_artifact` に基づき補助成果物へ分離されること
- 分離内容、非可逆性、fallback の有無が diagnostics または conversion report に記録されること

---

### 3.6 diagnostics 検証

対象：

- type
- severity
- message

検証内容：

- 変換時の問題が正しく記録されること
- unsupported parameter が diagnostics に記録されること
- fallback 実行結果が diagnostics または conversion report に記録されること
- Adapter 側補助成果物への分離結果が diagnostics または conversion report に記録されること

---

### 3.7 Round-trip前提検証

検証内容：

- Import → Export の連続処理が成立すること
- Export → Import の連続処理が成立すること
- 中間状態（SansaVRM）が整合性を維持すること

※詳細な往復一致検証は「06_ラウンドトリップテスト仕様.md」で定義する

---

### 3.8 MuJoCo Adapter API 境界検証

対象：

- SansaVRM-MuJoCo-Adapter
- SansaVRM 本体が提供する公開 API

検証内容：

- Adapter が SansaVRM 内部データ構造へ直接依存しないこと
- Adapter が Module / Connection / Slot / Property / custom parameter を公開 API 経由で取得すること
- Adapter が diagnostics / conversion report / 生成成果物メタデータを公開 API 経由で記録すること
- SansaVRM 本体が MuJoCo 実行ランタイムに依存しないこと

---

### 3.9 io_scope 出力先判定検証

対象：

- `io_scope = mjcf`
- `io_scope = adapter_artifact`
- `io_scope = both`
- `io_scope = preserve_only`
- `io_scope = unsupported`
- `io_scope = source_raw`

検証内容：

- `mjcf` は `mjcf_mapping` に基づき MJCF 出力対象となること
- `adapter_artifact` は `adapter_artifact` に基づき補助成果物へ分離されること
- `both` は MJCF と補助成果物の両方へ出力されること
- `preserve_only` は SansaVRM 内部保持のみとなり、MJCF と補助成果物へ出力されないこと
- `unsupported` は diagnostics に記録されること
- `source_raw` は解釈されず元情報として保持されること

---

### 3.10 mjcf_mapping 検証

対象：

- element
- attribute
- path
- direction
- value_conversion
- required_mujoco_version

検証内容：

- `direction = import` の場合、MJCF から SansaVRM へ取り込まれること
- `direction = export` の場合、SansaVRM から MJCF へ出力されること
- `direction = import_export` の場合、双方向に対応すること
- 対象 MuJoCo バージョン外の場合、fallback または diagnostics が発生すること

---

### 3.11 adapter_artifact 検証

対象：

- controller_config
- runtime_config
- conversion_report
- diagnostics
- external_metadata

検証内容：

- `adapter_artifact.artifact_type` に基づき出力先が決定されること
- `adapter_artifact.path` に基づき補助成果物内の配置が決定されること
- `required_adapter_version` の範囲外では fallback または diagnostics が発生すること

---

## 4. テスト方針

---

### 4.1 正常系

- 正しい外部データが正常に変換されること
- 出力がフォーマット仕様を満たすこと
- schema に基づき MJCF 出力対象と Adapter 側補助成果物対象が正しく分類されること

---

### 4.2 異常系

- 不正データが適切に検出されること
    - 必須要素欠落
    - 型不一致
    - 不正構造
- 変換不能部分が diagnostics に記録されること
- 未登録 custom parameter が検出されること
- `io_scope` と `mjcf_mapping` / `adapter_artifact` の不整合が検出されること
- 対象 MuJoCo バージョン非対応が検出されること

---

### 4.3 網羅性

以下を必ずカバーする。

- 全フォーマット（glTF / VRM / URDF）
- Import / Export 双方向
- 構造 / 接続 / 状態
- MuJoCo 標準連携対象としての Adapter API 境界
- `io_scope` 全値
- `mjcf_mapping` / `adapter_artifact`
- fallback 全値

---

### 4.4 再現性

- 同一入力で同一出力となること
- 同一 schema と同一 custom parameter に対して同一の出力先判定となること

---

## 7. 成功条件

以下を満たすこと。

- 変換結果が仕様と一致する
- ID が安定している
- diagnostics が期待と一致する
- source_raw が適切に保持される
- MJCF 出力対象が schema に基づいて判定される
- Adapter 側補助成果物対象が schema に基づいて判定される
- Adapter が SansaVRM 内部構造へ直接依存しない

---

## 8. 失敗条件

以下の場合は失敗とする。

- 変換結果が不正
- 構造が欠落
- ID が不安定
- diagnostics が不正
- schema に反して MJCF 出力対象を推測で判定している
- `io_scope` と `mjcf_mapping` / `adapter_artifact` の整合性が崩れている
- Adapter が SansaVRM 内部構造へ直接依存している

---

## 9. テストデータ管理

- フォーマット別に管理
- MuJoCo 連携テストでは custom parameter schema、adapter_artifact、conversion report のテストデータを管理する

例：

```text
tests/convert/
  gltf/
  vrm/
  urdf/
  mujoco/
    schema/
    adapter_artifacts/
    reports/
```
