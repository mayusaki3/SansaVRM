# ラウンドトリップテスト仕様

## 1. 目的

本仕様は、SansaVRM におけるラウンドトリップ（往復変換）の検証テスト仕様を定義する。

本仕様の目的は以下とする。

- 外部フォーマットと SansaVRM 間の変換が往復可能であることを確認する
- 変換における情報保持率を検証する
- 非可逆変換が仕様通りに扱われることを保証する
- ID安定性および構造整合性を検証する

---

## 2. テスト対象

以下のラウンドトリップを対象とする。

- glTF ⇄ SansaVRM ⇄ glTF
- VRM ⇄ SansaVRM ⇄ VRM
- URDF ⇄ SansaVRM ⇄ URDF

---

## 3. テスト分類

---

### 3.1 同一フォーマット往復

検証内容：

- 入力 → Import → Export → 出力
- 構造的同一性が維持されること

対象：

- glTF
- VRM
- URDF

---

### 3.2 異フォーマット経由

検証内容：

- glTF → SansaVRM → URDF → SansaVRM → glTF
- 異フォーマット間でも整合性が維持されること

---

### 3.3 構造整合性検証

検証内容：

- Module / Slot / Connection が一致すること
- State / Property が保持されること
- 不整合が発生しないこと
- property_type が保持されること
- context が保持されること
- connection_type が保持されること
- Module の親子関係（Connection構造）が保持されること

---

### 3.4 ID安定性検証

検証内容：

- 同一入力から同一IDが維持されること
- 往復後もIDが変化しないこと

---

### 3.5 非可逆変換検証

検証内容：

- 失われた情報が `source_raw` に保存されること
- `diagnostics` に warning が記録されること
- error とならないこと（仕様に従う）
- Property分類が失われないこと
- lossが発生した場合は diagnostics に記録されること
- lossy許容ケースでは、失われる情報が仕様上許容された範囲であること
- 例：URDF → SansaVRM → glTF で URDF 固有の joint / physical constraint 情報が完全復元できない場合
- 許容された loss は diagnostics に warning として記録されること
- 許容差分（tolerance）内の差異は成功とみなす

---

### 3.6 diagnostics 整合性

検証内容：

- 往復時の問題が正しく記録されること
- diagnostics の内容が一貫していること

---

## 4. テスト方針

---

### 4.1 正常系

- 往復変換が成立すること
- 出力が仕様に準拠すること

---

### 4.2 異常系

- 不正データが検出されること
- 往復不能なケースが適切に扱われること

---

### 4.3 網羅性

以下を必ずカバーする。

- 全フォーマット（glTF / VRM / URDF）
- 単純構造 / 複雑構造
- 状態あり / 状態なし

---

### 4.4 再現性

- 同一入力で同一結果となること

---

## 5. テストケース設計

各テストケースは以下を持つ。

- テストID
- テスト分類
- 入力フォーマット
- 入力データ
- 変換経路
- 期待結果
- 差分（diff）
- diagnostics

## 7. 成功条件

以下を満たすこと。

- 構造的同一性または許容差分内で一致する
- ID が安定している
- 非可逆情報が適切に扱われる
- diagnostics が期待と一致する

---

## 8. 失敗条件

以下の場合は失敗とする。

- 構造が崩れる
- ID が変化する
- 不整合が発生する
- diagnostics が不正

---

## 9. 差分評価

差分は以下で評価する。

- 構造差分（必須）
- 属性差分（許容あり）
- 非可逆差分（warning扱い）
- Property分類差分（必須）
- Connection差分（必須）

許容差分（tolerance）は以下とする：

- 浮動小数点誤差：±1e-6 以内
- 順序差分：順不同として扱う（配列順序が意味を持たない場合）
- ID順序：比較対象外（ID値自体は一致必須）
