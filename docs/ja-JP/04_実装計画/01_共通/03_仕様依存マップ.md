# 仕様依存マップ

## 1. 目的

本ドキュメントは、SansaVRM 仕様群の依存関係を整理し、以下を安全に実施するための基準を定義する。

```text
- ファイル並び替え
- 番号再配置
- 目次再編
- Traceability再構築
- Core / Extension 分離
```

---

## 2. 現在の問題

現在の仕様番号は、主に追加順で並んでいる。

そのため：

```text
- semantic依存順
- runtime依存順
- validator依存順
- adapter依存順
```

になっていない。

結果として：

```text
- 仕様探索性低下
- dependency不明瞭化
- traceability複雑化
- validator生成困難化
```

が発生している。

---

## 3. 基本方針

仕様群は以下のレイヤへ分離する。

```text
10 Core Semantic Layer
20 Preservation / Compatibility Layer
30 Data Model Layer
40 Runtime Integration Layer
50 Validation Layer
60 Import / Export Layer
70 Implementation / Roadmap Layer
```

依存方向は原則として：

```text
上位番号
↓
下位番号依存禁止
```

とする。

---

## 4. レイヤ定義

## 4.1 Core Semantic Layer

責務：

```text
- semantic定義
- semantic preservation定義
- roundtrip基準
```

対象候補：

```text
Core Semantic Definition
Semantic Preservation Matrix
RoundTrip Semantic Criteria
```

---

## 4.2 Preservation / Compatibility Layer

責務：

```text
- source_raw
- preservation
- compatibility
- approximation
- mapping
```

対象候補：

```text
Adapter Extension Property Specification
Format Compatibility Preservation Specification
Conversion Profile Specification
```

---

## 4.3 Data Model Layer

責務：

```text
- Geometry
- Rig
- Skinning
- Morph
- Animation
- Physics
```

対象候補：

```text
Geometry Rig Skinning Extension Specification
Morph Extension Specification
Animation Extension Specification
Physics Extension Specification
```

---

## 4.4 Runtime Integration Layer

責務：

```text
- MuJoCo integration
- URDF integration
- runtime binding
- runtime abstraction
```

対象候補：

```text
MuJoCo連携仕様
物理・制御メタモデル仕様
```

---

## 4.5 Validation Layer

責務：

```text
- validator
- diagnostics
- schema validation
- traceability
- regression
```

対象候補：

```text
Validator実装仕様
JSONスキーマ仕様
変換仕様
```

---

## 4.6 Import / Export Layer

責務：

```text
- VRM import/export
- FBX adapter
- MMD adapter
- glTF mapping
```

対象候補：

```text
VRM 0.x / 1.0 差分整理
VRM import詳細設計
```

---

## 4.7 Implementation / Roadmap Layer

責務：

```text
- roadmap
- migration
- implementation planning
```

対象候補：

```text
ロードマップ再整理
仕様依存マップ
```

---

## 5. 推奨依存方向

推奨依存方向：

```text
Core Semantic Layer
↓
Preservation / Compatibility Layer
↓
Data Model Layer
↓
Runtime Integration Layer
↓
Validation Layer
↓
Import / Export Layer
↓
Implementation / Roadmap Layer
```

---

## 6. 依存ルール

## 6.1 Core Semantic Layer

Core Semantic Layer は他 Layer に依存してはならない。

---

## 6.2 Preservation / Compatibility Layer

Preservation / Compatibility Layer は Core Semantic Layer のみに依存可能。

---

## 6.3 Data Model Layer

Data Model Layer は以下へ依存可能。

```text
- Core Semantic Layer
- Preservation / Compatibility Layer
```

---

## 6.4 Runtime Integration Layer

Runtime Integration Layer は以下へ依存可能。

```text
- Core Semantic Layer
- Preservation / Compatibility Layer
- Data Model Layer
```

---

## 6.5 Validation Layer

Validation Layer は全Layer参照可能。

ただし：

```text
semantic定義
```

を Validation Layer 側へ再定義してはならない。

---

## 6.6 Import / Export Layer

Import / Export Layer は以下へ依存可能。

```text
- 全仕様Layer
```

ただし：

```text
semantic定義変更
```

は禁止する。

---

## 7. 並び替え方針

ファイル並び替えでは以下を実施する。

```text
1. Layer単位へ再配置
2. semantic依存順へ並べ替え
3. validator依存方向修正
4. Runtime依存を末尾へ移動
5. import/exportを末尾側へ移動
```

---

## 8. 番号再配置方針

番号再配置では以下を推奨する。

```text
10xx Core Semantic
20xx Preservation / Compatibility
30xx Data Model
40xx Runtime Integration
50xx Validation
60xx Import / Export
70xx Implementation
```

例：

```text
1001_Core Semantic Definition
1002_Semantic Preservation Matrix
1003_RoundTrip Semantic Criteria

2001_Adapter Extension Property Specification
2002_Format Compatibility Preservation Specification
2003_Conversion Profile Specification

3001_Geometry Rig Skinning Extension Specification
3002_Morph Extension Specification
3003_Animation Extension Specification
3004_Physics Extension Specification
```

---

## 9. Traceability方針

Traceability は Layer 単位で管理する。

最低限以下を持つ。

```text
layer_id
spec_id
depends_on
validator_scope
runtime_scope
```

---

## 10. 今後の推奨作業順

推奨順序：

```text
1. 仕様依存整理
2. Layer定義固定
3. ファイル番号再配置
4. 目次再編
5. Traceability更新
6. validator参照更新
7. Import / Export実装再開
```

---

## 11. 結論

SansaVRM は現在、semantic integration platform としての中核仕様が揃いつつある。

今後は：

```text
- semantic依存方向
- compatibility責務
- runtime責務
- validation責務
```

を整理し、Layer化された仕様体系へ移行する必要がある。

その後、VRM / FBX / MMD / MuJoCo / URDF の import/export 実装を再開する。
