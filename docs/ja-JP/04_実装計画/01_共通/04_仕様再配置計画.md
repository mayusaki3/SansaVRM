# 仕様再配置計画

## 1. 目的

本ドキュメントは、SansaVRM 仕様群の Layer 化・番号再配置・目次再編を安全に実施するための移行計画を定義する。

本計画は以下を対象とする。

```text
- ファイル番号再配置
- Layer単位再編
- 目次再編
- 相互参照更新
- Traceability再構築
- validator参照更新
```

---

## 2. 現在の問題

現在の仕様番号は追加順となっている。

そのため：

```text
- semantic依存方向
- validator依存方向
- runtime依存方向
```

が反映されていない。

結果：

```text
- 探索性低下
- Layer責務不明瞭化
- validator自動生成困難化
- traceability複雑化
```

が発生している。

---

## 3. 再配置基本方針

再配置では以下を原則とする。

```text
1. semantic依存方向を固定する
2. Runtime依存を後段へ分離する
3. Compatibility責務を集約する
4. Validation責務を集約する
5. Import/Export責務を末尾へ移動する
6. 実装計画は最後段へ移動する
```

## 6. 再配置手順

推奨順序：

```text
1. Layer定義固定
2. 新番号定義
3. 目次新構成生成
4. 相互参照更新
5. Traceability更新
6. validator参照更新
7. 実ファイル移動
8. CI確認
```

---

## 7. 実施時の禁止事項

以下は禁止する。

```text
- semantic内容変更
- validator意味変更
- source_raw定義変更
- compatibility定義変更
- runtime責務変更
```

今回の再配置は：

```text
構造整理
```

のみを目的とする。

---

## 8. Traceability更新方針

Traceability は Layer 単位で再構築する。

最低限以下を持つ。

```text
layer_id
spec_id
spec_version
depends_on
validator_scope
runtime_scope
```

---

## 9. validator更新方針

validator 系は以下を更新する。

```text
- spec path
- traceability path
- schema path
- diagnostics path
- coverage matrix
```

semantic定義そのものは変更しない。

---

## 10. 目次再編方針

目次では以下を追加する。

```text
- Layer説明
- dependency説明
- runtime責務説明
- compatibility責務説明
```

また：

```text
追加順ではなくLayer順
```

で表示する。

---

## 11. 今後の推奨作業順

推奨順序：

```text
1. 仕様再配置計画固定
2. 新目次設計
3. Traceability再設計
4. validator参照更新
5. 実ファイル移動
6. CI確認
7. VRM import/export実装再開
8. FBX/MMD adapter実装
```

---

## 12. 結論

SansaVRM は現在、semantic integration platform として必要な主要仕様が揃いつつある。

今後は：

```text
- Layer構造
- dependency方向
- compatibility責務
- validation責務
```

を整理し、長期運用可能な仕様体系へ移行する必要がある。

その後、VRM / FBX / MMD / MuJoCo / URDF 実装を再開する。
