# ロードマップ再整理

## 1. 目的

本ドキュメントは、SansaVRM 開発において先行した MuJoCo validator / schema-driven validation platform の成果を一旦凍結し、SansaVRM Core および VRM I/O を優先するための再整理方針を定義する。

---

## 2. 現在の状態

現在の開発では、本来の優先順よりも MuJoCo validator / schema-driven validation platform が先行している。

本来の優先順は以下である。

```text
VRM I/O 完成
↓
RoundTrip保証
↓
Core Property安定化
↓
URDF
↓
MuJoCo
↓
Validator Platform
```

一方、実際には以下が先行している。

```text
MuJoCo validator / schema-driven validation platform
```

その結果、validator platform は強いが、VRM import / export が未完成である。

---

## 3. 完了済み成果

MuJoCo Schema-Driven Validation Platform は、PoC として以下を実装済みである。

```text
- schema-driven validator
- diagnostics
- conversion report
- adapter artifact separation
- fallback handling
- preserve_only / unsupported / source_raw
- traceability matrix
- traceability coverage analysis
- JSON Schema validation
- schema evolution compatibility checker
- GitHub Actions CI
- formal schemas
- traceability manifest
```

これらは PoC として十分な到達状態にあるため、現時点では追加拡張を一旦凍結する。

---

## 4. 現在の主要問題

### 4.1 VRM I/O 未完成

VRM 0.x / 1.0 import / export が未完成である。

特に以下を優先する。

```text
- humanoid
- expression
- spring bone
- lookAt
- meta
- extension passthrough
```

### 4.2 ドキュメント構造の不安定化

`docs/ja-JP/` 以下に、目次からリンクされていない文書が存在する可能性がある。

traceability を導入済みであるため、正式仕様集合が固定されない状態では、仕様・テスト・実装の紐づけが破綻する。

### 4.3 Validator Platform 先行問題

SansaVRM の本来の価値は、validator platform 単体ではなく、以下の統合である。

```text
VRM / URDF / MuJoCo 統合
```

そのため、validator platform は SansaVRM Core の意味論が安定した後に再開する。

---

## 5. 新ロードマップ

## Phase 0：ドキュメント再編

### 目的

正式仕様集合を固定し、traceability 対象を明確化する。

### 作業

1. `docs/ja-JP/目次.md` を正規化する。
2. 未リンク文書を棚卸しする。
3. 文書を以下に分類する。

```text
- 正式仕様
- Core Semantic
- 草案
- 実験
- validator専用
- adapter専用
- 廃止予定
```

4. traceability 対象仕様を固定する。
5. 旧階層と新階層が混在している場合、正式な配置先を決定する。

### 完了条件

- 目次から正式仕様へ到達できる。
- 正式仕様集合が一覧化されている。
- traceability 対象仕様が固定されている。
- 草案、実験、validator専用、adapter専用、廃止予定の文書が正式仕様と混在していない。

---

## Phase 1：VRM I/O 完成

### 目的

以下の変換で意味的欠落なしを成立させる。

```text
VRM 0.x / 1.0
↓
SansaVRM
↓
VRM 0.x / 1.0
```

### 方針

JSON 完全一致は要求しない。

重視するのは、意味的情報保持である。

### 重点対象

```text
- humanoid
- expression
- spring bone
- lookAt
- meta
- extension passthrough
```

---

## Phase 2：RoundTrip 保証

### 目的

以下の流れで意味的欠落なしを保証する。

```text
Import
↓
Export
↓
ReImport
```

### 作業

- namespace の安定化
- passthrough の安定化
- typed property の安定化
- semantic roundtrip test の作成
- import loss / export loss の分離

---

## Phase 3：Core Property 安定化

### 目的

SansaVRM Core の意味論と Property を安定化する。

### 作業

- Property formalization
- namespace stability
- migration policy
- version compatibility
- Core Semantic Definition の整備
- Semantic Preservation Matrix の整備

---

## Phase 4：URDF 強化

### 目的

VRM I/O と Core Property が安定した後、URDF 連携を強化する。

URDF は構造寄りのフォーマットであるため、VRM 完了後の Core 設計を前提に扱う。

---

## Phase 5：MuJoCo Runtime / Adapter

### 目的

現在の MuJoCo validator platform の成果を活用し、実際の MuJoCo runtime / adapter 連携へ進める。

### 不足対象

```text
- real MJCF emitter
- actuator emission
- sensor emission
- runtime mapping
- sysid integration
- HIL integration
```

---

## Phase 6：Validator Platform 再開

### 目的

SansaVRM Core の意味論および正式仕様集合が固定された後、validator platform の拡張を再開する。

### 対象

```text
- validator code generation
- schema bootstrap
- migration checker
```

### 注意点

仕様固定前に code generation を進めると、仕様変更時の作り直しが大きくなるため、Phase 6 まで後回しにする。

---

## 6. 凍結対象

以下は現時点で PoC として十分なため、一旦凍結する。

```text
tools/mujoco_schema_validator/
schemas/mujoco/schema_driven_validation/
traces/mujoco/schema_driven_validation/
```

凍結中は、以下を原則として行わない。

```text
- validator code generation の追加
- schema bootstrap の追加
- migration checker の高度化
- MuJoCo validator platform 単体の機能拡張
```

ただし、既存 CI を壊さないための保守修正は許可する。

---

## 7. 直近タスク

直近では以下を優先する。

```text
1. docs/ja-JP/目次.md の再整理
2. 未リンク仕様の棚卸し
3. 正式仕様セットの固定
4. Core Semantic 文書の作成
5. Semantic Preservation Matrix の作成
6. VRM I/O 実装への復帰
```

---

## 8. 判断基準

今後の作業優先度は以下で判断する。

| 優先 | 対象 | 判断 |
|---|---|---|
| 1 | VRM I/O | SansaVRM の入口として最優先 |
| 2 | RoundTrip | 意味的情報保持の保証に必要 |
| 3 | Core Property | 外部フォーマット統合の基盤 |
| 4 | URDF | 構造系フォーマット連携 |
| 5 | MuJoCo Runtime / Adapter | 物理・制御連携 |
| 6 | Validator Platform | 仕様固定後に再開 |

---

## 9. 結論

現在の MuJoCo validator / schema-driven validation platform は、PoC として十分に強い状態に到達している。

今後は validator platform の拡張を一旦凍結し、SansaVRM Core および VRM I/O を優先する。

これにより、SansaVRM の中心価値である外部フォーマット統合を先に成立させ、その後に validator platform を正式な品質保証基盤として再統合する。
