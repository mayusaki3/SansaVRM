# 実行基盤

## 1. 目的

本仕様は、SansaVRM における処理系の実行基盤および
各プラットフォームとの接続方式を定義する。

SansaVRM は複数環境（O3DE / Unity / VRChat / シミュレータ等）での
利用を前提とするため、処理ロジックを共通化しつつ、
各環境と適切に接続できる構造を必要とする。

---

## 2. 基本方針

- 中核処理は単一実装とする
- 実行環境依存処理は分離する
- 言語間インターフェースは安定したABIを採用する
- Runtimeとデータ構造は分離する
- glTFはデータ交換用途を担うフォーマットである

本仕様は主に SansaVRM Core および SansaVRM Runtime に関する定義を扱う。

---

## 3. SansaVRM Core（中核実装）

SansaVRM の中核処理は Rust により実装する。

### 3.1 対象処理

- メタモデル処理
- Slot管理
- モジュール構成管理
- Compatibility判定
- Rights統合
- Revenue計算
- SansaVRM Format との変換処理（シリアライズ / デシリアライズ）
- diagnostics生成
- 表情状態管理
- 状態切替管理
- 表現切替管理

---

## 4. インターフェース方針

中核処理は C ABI を正としたインターフェースを提供する。

### 4.1 理由

- C / C++ / C# から利用可能
- ABI安定性が高い
- 各環境での利用が容易

---

## 5. 各プラットフォーム接続

### 5.1 O3DE

- C++ ラッパーを介して接続
- Rust Core をネイティブライブラリとして利用

---

### 5.2 MuJoCo / シミュレータ

- C / C++ インターフェース経由で接続
- 物理シミュレーションとは独立して構成管理を行う

---

### 5.3 Unity（通常環境）

- C# からネイティブプラグインとして接続
- P/Invoke 等により Rust Core を呼び出す

---

### 5.4 VRChat

VRChat 環境ではネイティブコードの直接実行を前提としない。

- Udon / UdonSharp による制御を前提とする
- Rust Core は事前処理または外部処理として利用する
- Runtime 内では軽量な状態制御のみを行う

---

## 6. 処理分離

以下の責務を分離する。

### 6.1 Rust Core

- 判定
- 計算
- 構造処理
- 表情 / 外観切替判定
- 状態遷移判定

---

### 6.2 プラットフォーム側

- 表示
- 入力
- 物理処理
- UI

---

## 7. データと処理の関係

SansaVRM は以下の役割分担を持つ。

- SansaVRM Format  
  データ交換および保存のためのフォーマット（外部表現）  
  （glTF + extensions）

- SansaVRM Core  
  メタモデル処理および各種判定・計算を行う処理系

- SansaVRM Runtime  
  実行環境として、状態管理・イベント処理・外部連携を担う

各要素は独立して設計され、
フォーマットに処理ロジックを含めない。
