# 目的

## 1. 背景

従来のVRMは主にヒューマノイドモデルを対象として設計されており、
非人型モデル（ロボット、昆虫、クモ、抽象オブジェクト等）への対応は限定的である。

また、モデル構造、装着、権利管理、収益分配などが分断されており、
プラットフォームとしての拡張性に課題がある。

---

## 2. 目的

SansaVRM は、以下を実現するための統合モデル基盤である。

- あらゆる構造を持つモデルの表現（ヒューマノイド非依存）
- モジュール分割による柔軟な構成
- モデル・装着・権利・収益の統合管理
- Runtime における動的構成
- ワールドオブジェクトとの統合
- 経済圏の構築
- 表情、材質変化、テクスチャ変化を含む拡張表現の統合管理
- モジュールやアクセサリーの状態切替・着脱切替の統合管理

## 2.1 SansaVRM の構成要素

SansaVRM は単一の仕様またはフォーマットを指すものではなく、
以下の3つの構成要素からなる統合基盤として定義する。

- SansaVRM Format  
  モデル、モジュール、権利、状態などを保持するデータ表現形式。  
  glTF 拡張を含む外部交換フォーマットとして利用される。

- SansaVRM Core  
  メタモデル処理、構成管理、適合判定、権利判定、収益計算、
  状態切替などを担う中核処理系。Rust により実装される。

- SansaVRM Runtime  
  各プラットフォーム上で動作する実行環境。  
  SansaVRM Core を利用し、モデルの動的構成および状態管理を行う。

本仕様における「SansaVRM」は、特に断りがない限り、
上記3要素の総称として扱う。

SansaVRM Core が扱う内部メタモデルは、
SansaVRM Format としてシリアライズ可能である。

SansaVRM Format は、
SansaVRM Core におけるメタモデルの外部表現であり、
glTF をベースとした拡張形式として定義される。

---

## 3. スコープ

本仕様の対象範囲は以下とする。

- モデル構造定義
- モジュール構成
- 装着・接続
- 適合判定
- 権利・許諾制御
- 収益分配
- Runtime 動作
- 実行基盤およびインターフェース定義

---

## 4. 非スコープ

以下は本仕様の対象外とする。

- レンダリング方式詳細
- 物理演算詳細
- エンジン依存実装
- UI/UX設計
- ネットワーク同期仕様

---

## 5. 設計思想

### 5.1 非ヒューマノイド前提

すべてのモデルは構造ベースで定義される。  
ヒューマノイドはその一形態とする。

---

### 5.2 モジュール指向

モデルは複数のモジュールの組み合わせで構成される。

---

### 5.3 スロットベース構成

装着・機能・権利・収益はすべてスロットとして管理する。

---

### 5.4 動的構成

Runtime において構成変更が可能である。

---

### 5.5 分離設計

以下は独立して設計される。

- 構造
- 機能
- 権利
- 収益

---

## 6. SansaVRMの位置付け

SansaVRM は、フォーマット・処理・実行環境を分離した構造を持つ。

- SansaVRM Format  
  glTF をベースとした拡張形式として定義されるデータ交換フォーマット

- SansaVRM Core  
  フォーマット非依存の内部メタモデルを扱う処理系

- SansaVRM Runtime  
  Unity / O3DE / VRChat / シミュレータ等の実行環境

既存フォーマットとの関係は以下の通りとする。

- VRM → SansaVRM Format へ変換
- URDF → SansaVRM Core（Rig Slot）へ変換

glTF は SansaVRM Format の物理表現基盤として使用されるが、
SansaVRM Format 自体は glTF に依存しない論理フォーマットである。

glTF は内部構造ではなく、
SansaVRM Format の外部交換表現として使用する。

---

## 7. 想定ユースケース

- アバター（ヒューマノイド / 非ヒューマノイド）
- ロボットシミュレーション
- ワールドオブジェクト
- ゲームキャラクター
- デジタルツイン
- マーケットプレイス連携
- 表情切替を伴うアバター表現
- テクスチャ差し替えを伴う見た目切替
- アクセサリーの複数状態切替（表示 / 非表示 / 差し替え）

---

## 8. 成功条件

- 任意構造モデルの表現が可能
- モジュール単位で再利用可能
- 自動適合判定が可能
- 権利違反を防止できる
- 収益分配が自動化される
- 複数プラットフォームで利用可能

---

## 9. 将来拡張

- AI連携（自動生成・最適化）
- 自律構成モデル
- リアルタイム共有空間
- 分散型経済圏
